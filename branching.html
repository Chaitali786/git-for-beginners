Git Branching Guide

Branching helps minimise conflicts and isolate work from the main codebase. It also “quarantines” unreviewed code or potential bugs so they don’t affect the main branch. Every Git project starts with a single branch — usually called main (or master in older repositories).


Devs don’t work directly on main With multiple developers contributing simultaneously, the main branch can quickly become messy with conflicts.


Best practice

  • Each developer creates a separate branch for their work
  • They make commits and test their code inside that branch.
  • Once the work is complete and reviewed, the branch is merged back into main.


Benefits Best practice

  • Reduces conflicts on main.
  • Keeps untested or experimental code separate.
  • Makes collaboration cleaner and easier to manage.


Create branches. To create a branch, use the checkout command. The new branch will be an exact copy of the branch you were on when you ran the command. Typically, new branches are always created from the main branch

git git checkout -b branch-name - pull specific branch

git checkout branch-name // If you want to switch to an existing branch, omit -b

Once inside the new branch, you can add, commit, and push changes as usual. These changes will not affect other branches.

git checkout . // discards all un-staged changes in the current branch but keeps you on the same branch.


Merge main into your branch.

When you’ve finished working in your branch, before opening the PR, it’s good to update your branch and resolve conflicts. Since your branch was created from an earlier version of main, conflicts are possible. Using branches allows you to isolate these conflicts without affecting other branches.

  • Switch to the main branch and update it. This ensures your local main branch is up-to-date. git checkout main git pull
  • witch back to your feature branch and merge the latest main into it: git checkout your-branch git merge main
Any conflicts will appear in your branch, not in main. *** Remember: you merge other branches into your current branch, not the other way around.***


Resolve conflicts

If you encounter a conflict, the only way to resolve it is to manually decide which code to keep.

Often, you’ll want to keep all the code, the conflict usually happens because two sets of changes were made on the same lines.

VS Code provides tools to help with this, but when you’re learning version control, it’s a good idea to resolve conflicts manually.

Conflicts are marked in the file with symbols like <<<<<<<, =======, and >>>>>>>. You will need to decide to keep the current change or accept the incoming change, or keep both to resolve the conflict.

️Always check with your team members to ensure you don’t accidentally delete any important code
