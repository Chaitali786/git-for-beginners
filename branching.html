Git Branching

Branching helps minimize conflicts and isolate work from the main codebase. It also “quarantines” unreviewed code or potential bugs so they don’t affect the main branch. Every Git project starts with a single branch — usually called main (or master in older repositories). Devs don’t work directly on main With multiple developers contributing simultaneously, the main branch can quickly become messy with conflicts.

Best practice

  • Each developer creates a separate branch for their work
  • They make commits and test their code inside that branch.
  • Once the work is complete and reviewed, the branch is merged back into main.

Benefits Best practice

  • Reduces conflicts on main.
  • Keeps untested or experimental code separate.
  • Makes collaboration cleaner and easier to manage.

Create Branches

To create a branch, use the checkout command. The new branch will be an exact copy of the branch you were on when you ran the command. Typically, new branches are created from the main branch.

  • Create and switch to a new branch: 
    git checkout -b branch-name

    This creates and switches to a new branch called branch-name.

  • Switch to an existing branch: 
    git checkout branch-name

    Use this if the branch already exists. Omit the -b flag.

  • Discard all un-staged changes: 
    git checkout .

    This reverts un-staged changes in your current working directory without switching branches.

Once you're inside the new branch, you can add, commit, and push changes as usual. These changes will not affect other branches unless you merge them later.


Merge "main" into Your Branch

When you’ve finished working in your feature branch, it’s a good idea to update it before opening a pull request. Since your branch was likely created from an earlier version of main, there could be conflicts.

Using branches allows you to isolate these conflicts without affecting other developers or the main codebase.

  1. Switch to the main branch and update it:

    This ensures your local main branch is up to date with the latest changes.

    git checkout main 
    git pull
  2. Switch back to your feature branch and merge main into it: 
    git checkout your-branch
    git merge main

Any merge conflicts will appear in your branch — not in main. You can safely resolve them in isolation.

Note: You always merge other branches into your current branch, not the other way around.


Resolve Conflicts

If you encounter a conflict during a merge, the only way to resolve it is to manually decide which code to keep. Conflicts typically occur when changes have been made to the same lines of code in both branches.

Often, you may want to keep all the code — but you’ll need to make a conscious decision about which version (or combination) to retain.

VS Code provides helpful tools to assist with conflict resolution, but when you’re learning version control, it’s recommended to practice resolving them manually to better understand how Git works.

Conflicts are clearly marked in your file like this: <<<<<<< HEAD your changes ======= incoming changes >>>>>>> branch-name

You will need to choose between:

  • Keeping the current change
  • Accepting the incoming change
  • Keeping both changes and editing the file accordingly

Always check with your team members before resolving and committing, to ensure you don’t remove or overwrite important code by mistake.
